# ADR-004: AWS S3 for ion image storage

**Date:** 2026-03-27
**Status:** Accepted

## Context

Ion images are pre-rendered PNGs uploaded by scientists. A typical dataset contains 1,000–50,000 images at ~50–200 KB each. Images never change after upload (immutable). They must be served quickly to the annotation UI.

## Decision

Store ion images in **AWS S3**. Images are served to the frontend via **presigned URLs** with long TTLs, generated by the FastAPI backend on demand. The Nginx layer does not proxy image content — S3 handles it directly.

## Consequences

**Positive:**
- Decoupled from compute: EC2 instance can be resized or replaced without touching image data
- Durable (11 nines) with no operational effort
- Cheap: ~$0.023/GB/month. 50,000 × 100 KB = 5 GB ≈ $0.12/month
- Presigned URLs expire (default 1 hour) → no public bucket needed, no accidental exposure
- EC2 instance storage stays small

**Negative:**
- Additional AWS service to configure (IAM role, bucket policy)
- Presigned URL generation adds one FastAPI call before image loads (mitigated by React Query prefetching)
- Team must set `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_S3_BUCKET` env vars

## Abstraction

`backend/app/services/storage.py` is the single place where S3 is referenced. To swap to a different backend (e.g. Cloudflare R2, GCS, or local filesystem for dev), only this file changes.

## Alternatives Considered

- **Local filesystem on EC2:** Simple, but images lost if EC2 is terminated/replaced. Not durable.
- **Supabase Storage:** Would co-locate storage with DB. Supabase free tier has only 1 GB storage — insufficient. Pro plan is viable but adds cost. Kept as a future option.
